
# PROJECT CONTEXT (README.md):
The framework follows a 5-layer architecture to ensure scalability and parallel execution (via Pabot):
1. Global Lifecycle: Managed in common_resource.robot. Browser is launched per Suite; Tests are isolated via Contexts.
2. Locator Layer: resources/locators/ (using &{} dictionaries for namespacing).
3. Page Actions Layer: resources/keywords/ (atomic keywords for discrete interactions).
4. Functional Flow Layer: resources/flows/ (business sequences; no direct locators allowed).
5. Test Suite Implementation: tests/ (human-readable scenarios).

# MANDATORY RULE: LOCATOR FIDELITY
Do not "clean" or "optimize" locators. Use the exact selector strings generated by Playwright codegen.
- Translate Python methods to Browser library strings exactly.
- Example: page.get_by_role("button", name="Go") -> internal:role=button[name="Go"s].
- Example: page.locator("#frame").content_frame.get_by_role("btn") -> id=frame >>> internal:role=button.

# INSTRUCTION OPTION A: ADD NEW TEST CASE
If I provide a Playwright Python script and ask to "ADD A NEW TEST CASE":
1. Extract all selectors and add them to the correct &{} dictionary in 'resources/locators/'.
2. Create atomic keywords in 'resources/keywords/' for every interaction.
3. Create a high-level flow in 'resources/flows/' if the sequence is reusable.
4. Generate the Test Case in the .robot suite using the project's standard Setup/Teardown hooks.

# INSTRUCTION OPTION B: UPDATE EXISTING TEST CASE
If I provide a Playwright Python script, a Test Case Name, and a Suite Path, and ask to "UPDATE TEST CASE":
1. COMPARE: Check the new recording against the current implementation.
2. SELECTORS: Update the values in 'resources/locators/' with the new selectors. Keep existing variable names if the element's purpose is the same.
3. LOGIC CHANGES: If the recording includes new steps (e.g., handling a new popup), you MUST:
   - Extract the new locator.
   - Create a new atomic keyword in the 'Keywords' layer.
   - Insert this new keyword into the 'Flow' or 'Test Case' at the correct sequence point.
4. CLEANUP: Ensure the final test case remains human-readable and follows the established pattern.

# OUTPUT FORMAT:
Provide the output organized by file path:
- **File: resources/locators/...**
- **File: resources/keywords/...**
- **File: resources/flows/...**
- **File: tests/...**
